## [3.2.91] - 2026-05-28 *(Development milestone тАФ C03b Stage #4: Ids helper + UUID standardisation. **Closes C03b cluster.** Not promoted per `decisions/2026-05-26-release-cadence-all-clusters-then-ship.md`)*

### Added
- **`logic/Ids.kt`** тАФ single centralized source of entity IDs. `Ids.newId()` returns a fresh RFC 4122 v4 UUID (lowercase, hyphenated, 36 chars). Every new-entity-insert path in the Kotlin codebase now routes through here.

### Changed
- **C03b Stage #4 тАФ UUID standardisation.** Closes the audit's final ┬зC03b complaint: id formats drifted across the codebase. Survey at the start of this ship found:
  - 39 inline `UUID.randomUUID().toString()` calls across 17 files (the dominant pattern).
  - `PeopleScreen` used a custom `"P-${timestamp}-${hashSuffix}"` scheme тАФ collision-prone under fast inserts (millisecond clock + 4-digit hash тЖТ birthday-paradox collisions around ~100 inserts).
  - `MIGRATION_24_25` used SQLite-native `'P-' || lower(hex(randomblob(8)))` тАФ a third format, scoped to the one-time dedup pass.
- **Every Kotlin id-generation site** now calls `Ids.newId()`:
  - `FinanceRepository` (18 sites тАФ the densest), `FinanceViewModel` (2), `RecurringWorker`, `BankStatementImport`, `DummyDataSeeder`, `TransactionToRecurringTemplate`, and 11 dialog/screen files (`LendingDialog`, `DebtDialog`, `PaymentDialog`, `TransactionDialog`, `InvestmentDialog`, `ReportBugDialog` *[kept as-is тАФ it generates an 8-char display id, not an entity FK]*, `GoalScreen`, `InvestmentScreen`, `ProjectsScreen`, `SmartFlowWizardScreen`, `PeopleScreen`).
  - `PeopleScreen`'s `"P-${timestamp}-${hashSuffix}"` is now `Ids.newId()`.
- **Documented exceptions**:
  - `RecurringTransaction.id` default value stays `java.util.UUID.randomUUID().toString()` (the model layer must not depend on `logic/` тАФ would create a circular package dependency).
  - `MIGRATION_24_25`'s `'P-' || lower(hex(randomblob(8)))` stays тАФ SQLite has no built-in UUID generator, and the migration only runs once per user on the v24тЖТv25 upgrade. Two-format coexistence is bounded.
  - `ReportBugDialog`'s `UUID.randomUUID().toString().take(8).uppercase()` stays тАФ it's an 8-char display id for the bug-report dialog, not a database key.
  - The `"personal"` Project sentinel тАФ documented in `Ids.kt`'s KDoc as the one load-bearing non-UUID id in the schema. Treated as an immutable constant; never generated by `Ids.newId()`.

### Why this is the right shape
- This is **id hygiene**, not data rewrite. Existing legacy ids (`P-001`, `P-12345-6789`, `acc-cash`, etc.) stay valid because they're opaque FK targets тАФ Room compares them as strings, doesn't care about format. A destructive rename pass would cascade through every `projectId` / `ref` / `peopleId` / `fromAcctId` column тАФ high risk for a code-quality complaint.
- From this ship forward, every new entity gets a canonical UUID v4. The format-inconsistency stops growing; the legacy-format tail bounded to existing data.

### Data-integrity gate
- New JVM test: **`IdsDataIntegrityTest`** (4 cases):
  - Format: `UUID.fromString` parses it; version == 4; length == 36.
  - Canonical form: lowercase, hyphens at positions 8/13/18/23.
  - Distinct on consecutive calls.
  - **The collision-document case**: 10,000 tight-loop calls produce 10,000 distinct ids тАФ exactly the shape that broke the old `"P-${timestamp}-тАж"` scheme.
- Total: **296 tests across 26 classes**, 0 failures (was 292/25).

## [3.2.90] - 2026-05-28 *(Development milestone тАФ C03b Stage #3: Borrower/Debt peopleId FK with dedup; not promoted per `decisions/2026-05-26-release-cadence-all-clusters-then-ship.md`)*

### Added
- **C03b Stage #3 тАФ additive `peopleId` FK on `Borrower` and `Debt`, with phone-based dedup.** The audit's third structural complaint: lending/debt to the same person across multiple rows creates duplicate counterparties (free-text `name` + `phone` on each row, never aggregated). Now every loan/debt links to ONE `Person` keyed by phone:
  - **New columns**: `Borrower.peopleId`, `Debt.peopleId` (TEXT NOT NULL DEFAULT '').
  - **`MIGRATION_24_25`** тАФ three-step backfill:
    1. **Link to existing Person rows** by phone (captures implicit associations the user already created via the People screen).
    2. **Dedup-and-create** for unmatched non-empty phones тАФ `INSERT...SELECT GROUP BY phone` from `UNION ALL` of borrowers + debts. A phone that appears in both tables ends up with exactly one new Person and both rows link to it.
    3. **Re-link** every still-empty `peopleId` to the just-created Person rows.
    Idempotent: every step guards on `peopleId = ''` so re-running is safe.
  - **Empty-phone rows stay unlinked** by design. Dedup on name alone is unsafe (two "Ravi"s in the same project might be distinct people). The user links those manually via the People screen.
  - **Resolver-on-write**: new `FinanceRepository.findOrCreatePersonId(name, phone, projectId)` helper. Wired into `insertBorrowerWithSource` and `insertDebtWithDestination` so every new loan/debt deduplicates against existing People immediately тАФ no need to re-run the migration logic for normal use.
- **New DAO query** `getPersonByPhone(phone)` тАФ phone-keyed lookup, returns null for empty phone so empty-phone rows can never accidentally link to each other.
- **SyncManager** pulls `peopleId` on both `borrowers` and `debts` listeners. Legacy Firestore docs return `""` for the missing key.

### Why this is the right shape for Stage #3a (additive only)
- Stage #3a populates the FK without flipping any reads. Display still uses `Borrower.name` + `.phone` strings on every screen; the LendingScreen still keys on borrower rows, not Person rows.
- Stage #3b (deferred) will flip the reads: surface a "Sukesh тАФ 3 active loans" summary on LendingScreen, aggregate per-Person across the lending/debt tabs, etc. That's a UX rebuild, not a schema migration тАФ different ship.
- The migration's "dedup-by-phone" is the structurally-important move: from this ship forward, every new loan to the same phone joins the existing Person. The data shape stops drifting from the audit's target even before Stage #3b ships.

  - **`migrate24to25_leavesEmptyPhoneBorrowerUnlinked`** тАФ empty phone keeps `peopleId = ''` and no Person is invented (documents the "we don't dedup on name alone" rule).
- Migration-chain validation test includes `MIGRATION_24_25`.
- Total: **292 tests across 25 classes**, 0 failures (unchanged тАФ all new tests are instrumented, not JVM).

## [3.2.89] - 2026-05-28 *(Development milestone тАФ C03b Stage #1c: RecurringTransaction id mirror + RecurringWorker flip; not promoted per `decisions/2026-05-26-release-cadence-all-clusters-then-ship.md`)*

### Added
- **C03b Stage #1c тАФ additive `fromAcctId` / `toAcctId` on `RecurringTransaction`, plus `RecurringWorker` flip.** Stage #1b-1 closed the rename-safety loop for manually-recorded transactions but left `RecurringWorker.doWork` as the lone name-keyed write path because `RecurringTransaction` lacked an id mirror. This ship closes that gap:
  - **Entity**: `RecurringTransaction.fromAcctId` / `toAcctId` (TEXT NOT NULL DEFAULT '') тАФ mirrors the Stage #1a `Transaction` shape exactly.
  - **`MIGRATION_23_24`** backfills both columns by joining `recurring_transactions` against `accounts` on name. Identical SQL to `MIGRATION_22_23`, identical orphan semantics: rows whose name doesn't match keep id = `''` and route through the legacy name-keyed UPDATE in the worker.
  - **Resolver**: `RecurringTransaction.resolveAccountIdsWith(idByName)` тАФ sibling to the existing `Transaction.resolveAccountIdsWith` extension. Same decision rule (id wins, name fallback, idempotent).
  - **`FinanceRepository.insertRecurringTransaction`** resolves account ids on every save тАФ so a template the user creates today carries the id forward into every future auto-fire.
- **`RecurringWorker.doWork` is now rename-safe**:
  - The auto-generated `Transaction` it logs each month now copies the template's stored `fromAcctId` / `toAcctId` directly (instead of re-resolving by name, which could land on the wrong account if the user has both renamed and re-used a name since template creation).

### Why this matters
- Before this ship: rename "HDFC Bank" тЖТ "HDFC Salary" while you have a "Monthly Salary" recurring credit pointing at it тЖТ next month's auto-fire updates 0 rows (the worker's `updateAccountBalance("HDFC Bank", тАж)` doesn't match anything). The transaction gets logged but the balance silently doesn't move; users only notice when the net-worth math stops adding up.
- After this ship: the worker uses the stored `toAcctId` from the template, so the rename is invisible тАФ balance moves correctly regardless of how many times the account has been renamed since the template was created.
- This **closes the rename-safety loop for every balance-mutating write path that operates on a Transaction-shaped row**: manual entry (Stage #1b-1), and now auto-fires (this ship).

  - Empty-string id is harmless (it never matches; the repository's fallback routes those to the name-keyed query).

## [3.2.86] - 2026-05-28 *(Development milestone тАФ C03b Stage #1a: additive `fromAcctId` / `toAcctId` on Transaction; not promoted per `decisions/2026-05-26-release-cadence-all-clusters-then-ship.md`)*

### Added
- **C03b Stage #1a тАФ additive immutable account references on `Transaction`.** The biggest remaining audit complaint in ┬зC03b: `Transaction.fromAcct` / `.toAcct` store account-*names*, so an account rename silently orphans every row that referenced the old name (the 3.2.59 bug class). Stage #1a is the **additive** half of the fix:
  - **New columns**: `fromAcctId` and `toAcctId` (`TEXT NOT NULL DEFAULT ''`) on the `transactions` table. Empty string means "unresolved at write time".
  - **`MIGRATION_22_23`** backfills both columns by joining transactions against the current `accounts` table on `name`. Orphans (a `fromAcct` value with no matching `Account.name`) leave the id at `''` тАФ the migration must not invent an id; orphans surface to the 3.2.59 `OrphanTransactionsScanner` as before. Idempotent (`WHERE fromAcctId = ''` guard).
  - **Pure resolver** (`logic/TransactionAccountIdResolver.kt`): top-level `Transaction.resolveAccountIdsWith(idByName)` extension. Decision rule documented in the helper KDoc тАФ already-set ids survive untouched, blank names produce blank ids, resolvable names get their id, orphans stay empty. No coroutines, no Android deps тЖТ trivially testable.

## [3.2.83] - 2026-05-28 *(Development milestone тАФ XIRR for Lending + Debts; not promoted per `decisions/2026-05-26-release-cadence-all-clusters-then-ship.md`)*

### Added
- **`app.fynlo.logic.XirrCalculator`** тАФ pure-Kotlin Extended Internal Rate of Return solver. Newton-Raphson with damped step, 100-iter cap, 1e-7 tolerance. Returns `Double.NaN` for malformed / unsolvable input (single cashflow, all-positive or all-negative series, unparseable dates, non-convergence). `format(rate)` renders "12.4%" or "тАФ". Sign convention matches Excel / Sheets `XIRR`: negative amount = outflow from your pocket, positive = inflow.
- **XIRR on `CustomerDetailScreen` (Lending)** тАФ surfaces "Effective return (XIRR): X.X%" between payment-history header and the rows. Cashflows: `тИТprincipal` on loan date, `+payment.amount` on each Payment date, `+totalOutstanding` at today (the mark-to-today amount, so XIRR is meaningful for ongoing loans). Emerald-tinted Surface for positive return, red for negative.
- **XIRR on `DebtDetailScreen`** тАФ mirror image. Surfaces "Effective rate paid (XIRR): X.X%". Cashflows: `+principal` on debt date (received), `тИТpayment` on each DebtPayment date (paid back), `тИТtotalOutstanding` at today (still owed). Red-tinted Surface for positive rate (= cost to borrower), emerald in the rare case of a negative rate (heavily-prepaid interest-free debt or similar edge).
- **Investments intentionally skipped (no schema yet):** the `Investment` entity only carries `invested`, `currentVal`, `withdrawn`, `date` тАФ two effective cashflows, so XIRR mathematically collapses to CAGR. ### Tests
- `XirrCalculatorDataIntegrityTest` тАФ 13 cases: 2-cashflow doubling matches CAGR, 12-month monthly annuity yields ~18% (early-amortisation premium over nominal), irregular 90/180-day repayments converge in range, empty / single / all-positive / all-negative / bad-date inputs тЖТ NaN, same-day in+out is degenerate but doesn't crash, format helper renders one decimal % and dash sentinel.

### Why this shape
- **`+totalOutstanding` at today's date for both Lending + Debt**: standard treatment for open positions (Excel does the same with `XIRR` on a not-yet-closed flow series). The imputed mark-to-today gives the user "what rate AM I earning right now" rather than "what historical rate has cashed out so far".

### Why this shape
