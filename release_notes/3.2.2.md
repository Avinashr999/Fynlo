## 2. Migration / data-shape disclosure

On first launch after upgrading to 3.2.2, the app runs schema migration
`v15 → v16`. For every borrower or debt where the cumulative `paid`
value was greater than zero **and** no rows in the `payments` /
`debt_payments` tables existed for it (legacy data from versions before
the principal/interest split was tracked), the migration inserts **one
synthetic Payment row** with:

| Field | Value |
|---|---|
| `amount` | the borrower's / debt's existing `paid` total |
| `principal` | same as `amount` (best-effort default — actual split is genuinely unknown for legacy rows) |
| `interest` | `0` |
| `date` | the loan/debt start date (`loanDate`) — best available approximation |
| `type` | `"Legacy backfill"` |
| `notes` | `"Imported from legacy schema; actual repayment date unknown"` |

**User impact:** the loan/debt detail screen will show one new repayment
entry per affected record. Its amount is the user's existing `paid`
total (unchanged). Editing, splitting, or replacing the backfill entry
is permitted and recommended for users who remember the actual repayment
dates.

**Why this matters:** the same migration that fixes C01 also makes
`payments` / `debt_payments` the single source of truth for `paid`. The
backfill row is the safe-by-default record that lets future Recalculate
passes derive `paid` from `SUM(payments)` without losing legacy data.

**Halt criteria** (`RELEASE_PROTOCOL.md §6`): roll back immediately if any
of the following appear in the first 48 hours:

- Crash rate > 2 % attributable to 3.2.2.
- Any external user reports a P0 data-integrity issue (especially anything
  resembling lost `paid` values, lost repayment history, or mis-applied
  backfill rows).
- Negative rating change > 0.5 stars.
- Firebase outage > 2 h affecting sync.

If halt is triggered: follow `RELEASE_PROTOCOL.md §7` (roll back to 3.2.0
on Play Console, in-app banner if possible, postmortem within 7 days).

## 5. What 3.2.2 does NOT include

For visibility — features that have been discussed but are deliberately
**not** part of this release:

- **C02** (stale exports / auto-recalc before export). Now safe to ship
  on top of C01's fix, but is a separate cluster scheduled for the next
  3.2.x release.
- **C03a** (schema integrity additive fields — `schemaVersion`,
  `createdAt`, `projectId`, `userId` at backup root).
- **C05** (category bleed Income / Expense).
- **Instrumented migration test** (the missing Sprint 1 release gate).
  Adding `androidx.room:room-testing`'s `MigrationTestHelper` to
  `androidTest/` is its own small task; tracked as a follow-up.
- **Direct keystore-signed CI release** — `android.yml`'s release job
  remains an unsigned verification build. Signing happens locally for
  Play Store uploads, not in CI, per the slim Stage 1 spec.
