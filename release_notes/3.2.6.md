```
Smart defaults — Fynlo now remembers what you pick

• Add Transaction: the category for your last expense (or income) is
  pre-selected the next time you open the form. Typed a custom category
  like "Charity"? It comes back too — both the chip and the typed text.

  Add Transaction (chip selection instead of free text), and it
  remembers what you picked last time.

• Currency picker: on a fresh install, defaults to your device's
  locale (₹ for Indian devices, $ for US, € for European). After you
  pick once, your recent choices appear at the top of the list.
```

## 2. What changed under the hood

### Three surfaces wired to RecentlyUsedTracker

- **AddTransactionDialog** (already done in Stage 2, refined here with Stage 2.5): the `LaunchedEffect(isIncome)` block now has a three-case `when`:
  - No recency yet → both fields blank, user picks fresh.
  - Recent value is in `Categories.INCOME` / `Categories.EXPENSE` → chip pre-selected, `customCategory` cleared.
  - Recent value is a Custom-typed string → `selectedCategory = "Custom"` AND `customCategory = recent` together, so the text input below the chip row re-renders the user's previous typed value.

- **RecurringScreen → AddRecurringDialog** (`RecurringScreen.kt`): category input is now a `FlowRow` of `FilterChip`s instead of a free-text field. Same three-case prefill logic. Keyed off `FormIds.ADD_RECURRING` so its recency is isolated from one-off AddTransaction. Split by type (income vs expense) so a Salary recurring doesn't bleed into Expense recurring.

- **BudgetScreen → AddBudgetDialog**: chained-fallback prefill in `LaunchedEffect(Unit)`:
  1. `viewModel.suggestBudgetCategory()` — highest-spend EXPENSE category not yet budget-capped (the audit's load-bearing heuristic; NOT pure recency).
  2. If null, `viewModel.rememberLastBudgetCategory()` — recency fallback for when every spent-on category is already capped.
  3. If still null (fresh install, no spend), blank.
  Category input also converted from `LazyRow` + free-text to the C05-pattern chip-picker.

- **SettingsScreen → Currency picker** (`ExposedDropdownMenuBox`):
  - Initial selection: `viewModel.rememberLastCurrencyOrLocale(LocalConfiguration.current.locales[0])` — recency wins if any, otherwise the device locale's currency code (e.g., "INR" for `en_IN`, "USD" for `en_US`, "EUR" for `fr_FR`). Final fallback "INR" for locales without a country.
  - Menu content: conditional "Recently used" sub-header + ≤5 most-recent rows + `HorizontalDivider`, then the full alphabetical list from `CurrencyUtils.supported`. The recent section is hidden entirely when `observeRecentCurrencies()` emits an empty list (fresh install).
  - Each `DropdownMenuItem` writes to BOTH `UserPreferences.setDefaultCurrency` (canonical pref) AND `viewModel.recordCurrency(code)` (recency layer).

### Pure-function helper extracted for the budget heuristic

`app/src/main/java/app/fynlo/data/BudgetSuggestion.kt` — `object BudgetSuggestion.suggest(cappedCategories: Set<String>, expenseAnalytics: Map<String, Double>): String?`. Self-contained, no Android dependencies, no coroutines. The ViewModel's `suggestBudgetCategory()` is a thin delegate over it, so the heuristic is unit-testable without a Hilt + Room + StateFlow harness.

## 3. Migration / data-shape disclosure

**None.** 3.2.6 is a pure UI / behaviour milestone. No schema migration, no backup schema bump, no field rename. `@Database` version stays at `17`, backup `schemaVersion` stays at `2`, `RecentlyUsedSnapshot` JSON shape stays as defined in Stage 1.

The DataStore key `recently_used` now accumulates entries in additional slots (`add_recurring → category_income`, `add_recurring → category_expense`, `add_budget → category_expense`, `settings_currency → currency`), but the JSON envelope is unchanged from Stage 1's `RecentlyUsedSnapshot` definition — empty slots default to an empty list, so older installs that skip directly to 3.2.6 from a pre-C04 version read the empty snapshot correctly.

## 5. What 3.2.6 does NOT include

For visibility — features explicitly **not** part of this milestone:

- **C04 Stage 3 grouped-picker UX in chip surfaces.** Only the currency dropdown got the "Recently used" sub-group rendering. Chip-based pickers (AddTransaction, Recurring, Budget) don't need this since `FilterChip` layout already shows every option equally; prefill alone is the C04 win there. Documented this rationale in `FinanceViewModel.observeRecentCurrencies` doc comment.
- **C06 / C07** (FAB overlap + duplicate FAB). Next P1 Sprint 2 pair after this.
- **C08** (number formatting), **C09** (UTF-8 in dialogs), and the rest of the P1 backlog. Per release-cadence ADR they all ship together at C22 closure.
- **Compose UI tests** for the new picker behaviours. Blocked on INF05 (instrumented tests in CI). Data layer is fully covered by the 71-case data-integrity gate.
