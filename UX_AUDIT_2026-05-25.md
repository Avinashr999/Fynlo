## 1. The one finding that overrides everything

**Recalculate Balances destroys user data.**

When tapped, it deletes the `paid` field from every borrower and debt record. ₹54,000 of repayment history evaporated in the audit's test data. The XLSX export's "Loan Repayments" and "Debt Repayments" sheets are always empty because Fynlo doesn't store individual payments as records — only as a cumulative `paid` field on the parent. Wiping the field wipes the history.

This is **C01** in the cluster list and **P0-blocker #1** in the priority list.

**Recommendation: Do NOT ship 3.2.1 to Play Store production until C01 is fixed.**

The cold-start performance win from the baseline profile is real (~10% faster) and worth shipping — but not alongside a regression that nukes payment history when users tap a button labeled "Fixes account balances that got out of sync."

Ship 3.2.1 via Play Store internal testing only. Land the C01 fix as 3.2.2 and promote that to production instead.

## 2. The 22 root-cause clusters

### C01 — Recalculate Balances data destruction

**Raw points:** 346, 347, 348, 349, 350, 367
**Severity:** Critical data loss
**Priority:** P0 (ship-blocker)
**Effort:** L (3-5 days)
**Sprint:** 1

**Root cause:**
1. Payments are stored as a cumulative `paid: Double` field on the parent loan/debt record, not as individual payment records.
2. The Recalculate flow rebuilds loan/debt state from interest accrual without preserving the `paid` field.
3. `updatedAt` timestamps don't update when the destruction happens, breaking sync conflict resolution.

**Fix:**
1. **Schema change:** Add `loanRepayments[]` and `debtRepayments[]` collections. Each repayment is a record with `id`, `parentLoanId`/`parentDebtId`, `amount`, `date`, `note`, `createdAt`. Derive cumulative `paid` from the sum.
2. **Migration:** Backfill existing `paid` fields into single repayment records dated `loanDate` (best available approximation).
3. **Recalculate rewrite:** Compute outstanding = principal + accrued interest − sum(repayments). Never touch repayment records.
4. **`updatedAt` correctness:** Update timestamp on every actual mutation.
5. **Guard:** Add a unit test that runs Recalculate against a fixture with known `paid` values and asserts the sums are preserved.

**Impact on other clusters:** Unblocks C02 (stale exports), C03 (sync correctness).

### C02 — Stale exports & no auto-recalc

**Raw points:** 331, 332, 333, 334, 343, 344, 345
**Severity:** Data integrity
**Priority:** P0
**Effort:** M (2-3 days)
**Sprint:** 1

**Root cause:** Recalculate is manual-only. PDF, JSON, and XLSX exports use whatever in-memory state exists, which may be days or weeks stale. The PDF generated pre-recalc showed ₹268,081 net worth; post-recalc showed ₹241,663 — same data, ₹26K difference, no warning.

**Fix:**
1. **Auto-recalc on app launch.** Run once per day max (debounce on `lastRecalcAt < today.startOfDay`).
2. **Auto-recalc before every export.** PDF, JSON, XLSX must invoke recalc as a pre-step.
3. **Surface `lastRecalcAt` in UI.** Small subtitle on Dashboard near Net Worth: "Last updated 2 min ago".
4. **Show before/after on manual recalc.** When user taps Recalculate in Settings, show a result dialog: "Net worth: ₹268,081 → ₹241,663. Interest accrued: ₹85,651."
5. **Embed timestamp in exports.** PDF header includes "Recalculated: 25 May 2026, 9:07 AM."

**Dependency:** Requires C01 (otherwise auto-recalc destroys data daily).

**Stage 1 landed 2026-05-26 (fixes #1, #2 of the five):**

- New `app.fynlo.data.RecalcCoordinator` (`@Singleton`, Hilt-injected) wraps `FinanceRepository.recalculateAllBalances()` and stamps `UserPreferences.lastRecalcAt` after each successful run. Exposes a pure `shouldRecalcOnLaunch(lastRecalcAt, now, zone)` predicate on the companion (testable without Robolectric — calendar-day boundary in the user's local zone, NOT a 24h sliding window).
- `FynloApplication.onCreate` post-init coroutine now calls `recalcCoordinator.runIfStaleOnLaunch()` — fire-and-forget; failure is non-fatal.
- `FinanceViewModel` exports (`exportToCSV`, `exportToPDF`, `exportAllData`, plus a NEW `exportToXLSX`) all call `recalcCoordinator.runAndStamp()` before generating output. XLSX was previously skipped because the launcher called `ExcelExportUtility.generateFullBackup` directly — now routed through the ViewModel for consistency.
- `SettingsScreen` PDF launcher wrapped in `scope.launch(Dispatchers.IO)` (since `exportToPDF` is now suspend).
- `AutoRecalcDataIntegrityTest` (8 cases) covers the debounce predicate at every interesting edge: first-launch (lastRecalcAt = 0), midnight rollover, same-day-twice, local-vs-UTC zone, week-long gap, exact-equality-with-now, and a from-first-principles arithmetic check against `LocalDate.atStartOfDay`. Picked up by the `checks.yml` data-integrity gate (matches `*DataIntegrity*` and `*Recalculate*`).

**Stage 2 landed 2026-05-26 (fixes #3, #4, #5 — C02 fully CLOSED):**

- **Dashboard "Last updated X ago" subtitle** — `HomeScreenModern` consumes `UserPreferences.lastRecalcAt(...).collectAsState(initial = 0L)` and renders below the hero net-worth number using Android's `DateUtils.getRelativeTimeSpanString` (yields "2 min ago" / "yesterday" / "3 days ago" with appropriate granularity). Shows "Not recalculated yet" when `lastRecalcAt == 0L`.
- **Before/after AlertDialog on manual Recalculate** — `FinanceViewModel.recalculateAllBalancesCapturingDelta()` captures `financialSummary.value` before, suspending-calls `runAndStamp()`, then waits up to 500 ms for the next `FinancialSummary` emission different from the pre-snapshot (`withTimeoutOrNull`). Returns a `RecalcDelta` data class with `netWorthChange` / `receivablesChange` / `cashChange` / `investmentsChange` + an `isNoOp` predicate. `SettingsScreen`'s recalc handler shows the dialog with `before → after` for net worth plus signed deltas for the other three; if `isNoOp` (the common post-C01 case), the dialog shows a "your data was already up to date" message instead. The old fire-and-forget Toast is gone.
- **Timestamp in exports** — `ExportUtility.generatePDF` and `ExcelExportUtility.generateFullBackup` both accept a new `lastRecalcAt: Long = 0L` parameter. PDF: a `Recalculated: <date>` line under the existing `Generated:` line (or `—` if `lastRecalcAt == 0L`). XLSX: a new top-of-file `Metadata` sheet with `Generated` / `Recalculated at` / `Export type` rows. `RecalcCoordinator.runAndStamp()` was refactored to **return** the stamped time so the ViewModel exports don't need a separate DataStore read.

**Net effect:** the audit's quoted scenario ("PDF generated pre-recalc showed ₹268,081; post-recalc showed ₹241,663 — same data, ₹26K difference, no warning") is no longer reachable. The PDF can't show stale numbers because exports recalc first, the dashboard tells you when the data was computed, and tapping Recalculate explicitly shows the delta instead of silently mutating things.

### C03 — Schema integrity & sync correctness

**Raw points:** 276, 277, 278, 279, 280, 281, 282, 283, 284, 285, 286, 287, 289, 290, 291, 296, 297, 301
**Severity:** Data integrity
**Priority:** P0 / P1 mix
**Effort:** XL (1-2 weeks across the cluster)
**Sprint:** 2-3

**Root cause:** The data layer was built incrementally without a schema discipline. As a result:
- No `schemaVersion` / `appVersion` / `exportedAt` at backup root → restore can't migrate forward.
- Transactions reference accounts by `name` not `id` → renaming "Cash in Hand" breaks every historical transaction.
- `updatedAt` is epoch ms; `date` is ISO string → inconsistent within same record.
- Borrowers use `type`, Debts use `intType` → same concept, two field names.
- No `projectId` anywhere → "Manage Projects" feature has nothing to scope.
- Borrower phone is free-text, not FK to a `people[]` record → same person stored 3x for 3 loans.
- One transaction has `category: "Expense"` literal → default category bug.
- No `createdAt` field anywhere → can't tell when records originated.
- No data integrity hash / checksum → corrupted file restores silently.
- Mixed ID strategy (`acc-cash` vs UUIDs) → collision risk on multi-device merge.

**Fix — staged:**

**Stage 3a (Sprint 2) — additive, non-breaking: ✅ FULLY CLOSED 2026-05-26.**

1. ✅ **Done 2026-05-26 (Stage 1).** Backup root carries `schemaVersion = 2`, `appVersion = BuildConfig.VERSION_NAME`, `exportedAt = Instant.now().toString()`, `userId` (caller-provided, blank if signed out), `deviceName = Build.MODEL`. All five fields are populated by `FinanceRepository.getAllDataAsJson()`.
2. ✅ **Done 2026-05-26 (Stage 2).** `MIGRATION_16_17` adds `createdAt INTEGER NOT NULL DEFAULT 0` to `flow_templates`, `investment_valuations`, `net_worth_snapshots`, `recurring_transactions`. Three of the four are backfilled from `updatedAt`; `net_worth_snapshots` (no `updatedAt`) is backfilled from `strftime('%s', date) * 1000`. Entity classes updated to add the field.
3. ✅ **Done 2026-05-26 (Stage 2).** `MIGRATION_16_17` adds `projectId TEXT NOT NULL DEFAULT 'personal'` to `investment_valuations`, then backfills from the parent investment's `projectId` (orphan valuations fall back to `'personal'`).
4. ✅ **Done 2026-05-26 (Stage 1).** `contentHash` is a SHA-256 (lower-case hex) over the canonical JSON form of `BackupData` with `contentHash` set to `""`. Computed at export, embedded as the last step before serializing. Verified at import via `BackupIntegrity.check()` — restore refuses to proceed on hash mismatch (`IllegalStateException` thrown before `db.withTransaction` opens). Forward-compat: backups with `schemaVersion > 2` are also refused with `UnsupportedVersion`. Backwards-compat: v1 legacy backups (no metadata, no hash) are accepted unconditionally. `BackupDataIntegrityTest` covers all 10 paths.
5. ✅ **Done 2026-05-26 (Stage 2).** NEW `app.fynlo.data.TransactionValidator.sanitize(Transaction)` rewrites the three forbidden literal categories (`"Expense"` / `"Income"` / `"Transfer"`) to `"Uncategorized"`. Applied at both write sites: `FinanceRepository.insertTransaction` and the new-side of `editTransaction`. Matching is case-sensitive and exact — `"Income Tax"` and `"Expense Reimbursement"` pass through unchanged. A one-shot `UPDATE transactions SET category = 'Uncategorized' WHERE category IN ('Expense', 'Income', 'Transfer')` inside `MIGRATION_16_17` fixes any existing rows produced by the old buggy dropdown. `TransactionValidatorDataIntegrityTest` covers the contract (9 pure-function cases including idempotency, exact-match-only, fast-path no-allocation when input is valid).

**Sprint 1+2 milestone:** With C01 + C02 + C03a all closed, three of the four P0 ship-blockers (C01 / C02 / C03a) are done. The Sprint 1 audit estimate was C01-only; we landed C01 + C02 + C03a in the same window. C05 (category bleed Income/Expense) is the one remaining P0.

**Sprint 2+ remaining:** Stage 3b (breaking migrations — account-name → account-ID, `intType` unification, peopleId/contact linkage, UUID standardization) is unchanged from the original audit plan — Stage 3a's additive groundwork doesn't unblock or unblock Stage 3b directly.

**Stage 3b (Sprint 3) — breaking, requires migration:**
1. Convert `txn.fromAcct` / `txn.toAcct` from name strings to account IDs. Migration: lookup by name on import, store ID going forward.
2. Standardize `intType` everywhere (rename `borrowers[].type` to `borrowers[].intType`).
3. Move `borrower.phone` to a separate `people[]` record. Borrower references `peopleId`. Migration: dedupe phones, create people records, repoint.
4. Standardize on UUIDs across all collections. Migration: `acc-cash` → new UUID, update all references.

Each stage ships independently. Stage 3a is fully backwards-compatible. Stage 3b requires a versioned migration script that runs once at upgrade.

**Acceptance:** Round-trip test — export, wipe, restore, diff. Result must be byte-identical for all post-stage-3b backups.

### C04 — Smart defaults system (the missing system) ✅ CLOSED 2026-05-27 in 3.2.6 (smoke-follow-up patch in 3.2.7)

**Raw points:** 14, 15, 16, 167, 168, 213, 215, 216, 229, 230, 237, 263
**Severity:** UX friction, daily
**Priority:** P1
**Effort:** M (3-5 days) — actual: ~1 session (Stages 1+2 plus Stage 3 + 2.5 closure)
**Sprint:** 2 (first P1 cluster closed)

**Root cause:** Every form starts blank or with a wrong default. User taps + 30 times a day. Multiplied across forms (Add Transaction, Add Recurring, Add Savings, Set Category Limit, currency picker, account picker), this is the single biggest source of friction.

Specific missing defaults:
- Add Transaction: no default category (or wrong category, see C05). Should default to user's most-recently-used category for the current type.
- Add Transaction: no default account. Should default to most-recently-used.
- Add Recurring: no default category, no default account. Same fix.
- Set Category Limit: no default category. Should default to user's highest-spend uncapped category.
- Settings → currency: defaults to INR regardless of locale. Should default to system locale on first install.
- Recently-used should also appear at top of pickers (currency picker has no "recent" group).

**Fix:**
1. Build a `RecentlyUsedTracker` with per-form, per-field memory.
2. On every form open, prefill the field with `recentlyUsed.last(formId, fieldId)` or fall back to system locale / heuristic.
3. Picker dropdowns sort: recently-used (top, ≤5 items) → alphabetical (everything else).
4. Migrate to using `system_locale_default` for currency on fresh install.

**Stage 1 landed 2026-05-26 (data abstraction only — no UI wiring yet):**

- NEW `app.fynlo.data.RecentlyUsedTracker` (`@Singleton`, Hilt-injected). API: `suspend record(formId, fieldId, value)` / `suspend last(formId, fieldId): String?` / `observeTopN(formId, fieldId, n = 5): Flow<List<String>>`. Plus `FormIds` / `FieldIds` companion objects with the canonical slot constants (`ADD_TRANSACTION` / `ADD_RECURRING` / `ADD_BUDGET` / `SETTINGS_CURRENCY` and `CATEGORY_INCOME` / `CATEGORY_EXPENSE` / `FROM_ACCOUNT` / `TO_ACCOUNT` / `CURRENCY`) so call sites can't typo into orphan slots. Category is split by type (`CATEGORY_INCOME` vs `CATEGORY_EXPENSE`) so the same Income/Expense boundary C05 just enforced isn't bypassed via recency leakage.
- NEW `app.fynlo.data.RecentlyUsedLogic` — pure-function helpers (`add` / `last` / `topN` / `keyOf`). All dedup-and-cap semantics live here, independent of DataStore, so unit tests cover the contracts without Robolectric. `MAX_ENTRIES = 5` per the audit's "≤5 items" group size.
- NEW `app.fynlo.data.model.RecentlyUsedSnapshot` / `RecentEntry` — serializable types stored as a single JSON blob under `UserPreferences.RECENTLY_USED`. Single Preferences key keeps the dataset (≤5 entries × ~10 slots) addressable without a new Room table or migration. Forward-compat is the kotlinx-serialization default-values story.
- `UserPreferences.recentlyUsed(ctx): Flow<RecentlyUsedSnapshot>` + `editRecentlyUsed(ctx, mutate)` — DataStore reader / atomic editor. Fault-tolerant decode: a corrupted blob yields an empty snapshot instead of crashing.
- `RecentlyUsedDataIntegrityTest` — 12 pure-function cases. Picked up by the `*DataIntegrity*` CI gate filter. Gate count: **39 → 51** across 6 classes.

**Stage 2 landed 2026-05-26 (AddTransactionDialog category prefill + record):**

- `FinanceViewModel` injected with `RecentlyUsedTracker`. Two new methods: `suspend rememberLastTransactionCategory(isIncome): String?` and `recordTransactionCategory(isIncome, category)` (fire-and-forget). Both internally route to `FormIds.ADD_TRANSACTION` and the type-split `FieldIds.CATEGORY_INCOME` / `CATEGORY_EXPENSE` slots — recency does NOT leak across the same Income/Expense boundary C05 enforces on the chip list.
- `AddTransactionDialog` signature extended with two **optional** lambdas with no-op defaults: `rememberLastCategory: suspend (Boolean) -> String?` and `onRecordCategory: (Boolean, String) -> Unit`. Defaults keep the dialog testable/previewable without a ViewModel; production call sites bind viewModel-method-references.
- The C05 `LaunchedEffect(isIncome) { selectedCategory = "" }` is replaced with a recency-aware version: on every toggle flip (and initial open), it asks `rememberLastCategory(isIncome)` and pre-selects the answer **if** it's in the curated chip list for the new type. Falls back to blank for the no-recent case OR for custom values (e.g., the user's last "Charity" pick) — Stage 2.5 will handle custom-value re-prefill.
- The dialog's submit handler extracts the final category once (collapsing the previous double-evaluation of the `if (selectedCategory == "Custom") customCategory else selectedCategory` ternary), calls `onRecordCategory(isIncome, finalCategory)`, then `onConfirm(txn)`. Records the **final** value (e.g., "Charity"), not the chip-list sentinel "Custom".
- Wired into all four call sites: `Navigation.kt:202`, `SpendScreen.kt:71`, `HomeScreenModern.kt:82`, `HomeScreen.kt:54`. All four use `viewModel::rememberLastTransactionCategory` / `viewModel::recordTransactionCategory` as the lambda values.

**Stage 2.5 landed 2026-05-27 (custom-value re-prefill in AddTransactionDialog):** the Stage 2 `LaunchedEffect(isIncome)` silently dropped Custom-typed recency values (e.g., user-typed "Charity"). Replaced with a three-case `when`: null → blank both fields; recent in chip list → chip + clear customCategory; recent NOT in list → `selectedCategory = "Custom"` AND `customCategory = recent` together, so the typed text re-renders below the chip row. Folded into the 3.2.6 closure commit (same dialog, same cluster).

**Stage 3 landed 2026-05-27 (three remaining picker surfaces) — three parallel general-purpose agents, one screen each, zero merge risk:**

1. **`RecurringScreen.AddRecurringDialog`** — converted free-text category `OutlinedTextField` to the C05-pattern `FlowRow` of `FilterChip`s from `Categories.INCOME` / `Categories.EXPENSE` + trailing "Custom" chip + conditional custom-name input. Same three-case `LaunchedEffect(isIncome)` prefill logic. `viewModel.rememberLastRecurringCategory(isIncome)` reads on open; `viewModel.recordRecurringCategory(r.type == "Income", r.category)` writes on submit. Keyed off `FormIds.ADD_RECURRING` (isolated from one-off AddTransaction). Split by type so Income recurring recency doesn't bleed into Expense. **This also closes the C05 "Out-of-scope: RecurringScreen still free-text" deferral.**

2. **`BudgetScreen.AddBudgetDialog`** — chained-fallback prefill in `LaunchedEffect(Unit)`: (a) `viewModel.suggestBudgetCategory()` — the audit's **highest-spend-uncapped** heuristic, NOT pure recency; (b) `viewModel.rememberLastBudgetCategory()` fallback when every spent-on category is already budget-capped; (c) blank on fresh install with no expenses. Category input also converted from `LazyRow` + free-text to the C05 chip pattern.

3. **`SettingsScreen` currency picker (`ExposedDropdownMenuBox`)** — initial selection via `viewModel.rememberLastCurrencyOrLocale(LocalConfiguration.current.locales[0])` — recency wins if any, otherwise `Currency.getInstance(locale).currencyCode` (e.g., "INR" for `en_IN`, "USD" for `en_US`, "EUR" for `fr_FR`), with "INR" final fallback for locales without a country. Menu renders a conditional "Recently used" sub-header + ≤5 rows + `HorizontalDivider` above the full alphabetical list from `CurrencyUtils.supported`. Hidden when `observeRecentCurrencies()` is empty (fresh install). Each `DropdownMenuItem` writes to both `UserPreferences.setDefaultCurrency` (canonical pref) AND `viewModel.recordCurrency(code)` (recency layer).

**Pure-function extraction:** the budget heuristic lives in `app.fynlo.data.BudgetSuggestion.suggest(cappedCategories, expenseAnalytics)` — no Android, no coroutines, no Hilt. `FinanceViewModel.suggestBudgetCategory()` is a thin delegate. Testing the helper directly avoided a Hilt + Room + StateFlow harness for what is essentially "given two collections, pick a string."

**Tests added in Stage 3 (both match `*DataIntegrity*` so they run in the CI gate):**
- `BudgetSuggestionDataIntegrityTest` (12 cases): empty inputs, fully-capped state, mixed capped/uncapped (load-bearing), tie-break determinism, blank-category filter, zero/negative-spend filters, capped-set semantics.
- `CurrencyPickerOrderDataIntegrityTest` (8 cases): `buildCurrencyPickerOrder(recent, full)` helper for the dropdown's flat-dedup display order — empty-recent passthrough, recents-leading, cross-list dedup, recent-order preservation (most-recent-first, NOT alphabetical), full-list order preservation, blank stripping, intra-recent dedup, union-size invariant.

**Gate count: 51 → 71 across 8 classes. All pass.**

**Decision documented (in `FinanceViewModel.observeRecentCurrencies` doc comment):** "Recently used" group rendering only applies to the currency dropdown. Chip-based pickers (AddTransaction, Recurring, Budget) don't need it because `FilterChip` layout already shows every option equally — prefill alone is the C04 win there.

**Acceptance — all met:**
- ✅ Open Add Transaction → expense → Food pre-selected (after one prior use). Custom "Charity" pick re-renders both the Custom chip and the typed text. (Stage 2 + 2.5)
- ✅ Open Add Recurring → Monthly Rent → Rent pre-selected (chip picker now, not free text). (Stage 3)
- ✅ Open Add Budget → suggested chip is the user's highest-spend uncapped category. (Stage 3, heuristic)
- ✅ Fresh-install Settings → Currency picker → device-locale currency pre-selected; after picking, "Recently used" group appears above the alphabetical list. (Stage 3)
- ✅ No blank dropdowns on common forms.

### C05 — Category bleed across Income/Expense ✅ CLOSED 2026-05-26

**Raw points:** 11, 166, 230, 231
**Severity:** Data integrity (causes wrong category logging)
**Priority:** P0
**Effort:** S (1 day) — actual: ~1h
**Sprint:** 1

**Root cause:** The Add Transaction sheet's category chips don't swap when the Income/Expense toggle flips. Income tab shows Food/Rent/Fuel/Shopping (expense categories); Expense tab shows Salary/Investment/Lending (income categories). User picks wrong category, transaction is mis-tagged, P&L is wrong.

**Fix as landed:**
1. ✅ NEW `app.fynlo.data.Categories` — `INCOME` (8 entries), `EXPENSE` (13 entries), `TRANSFER` (1 entry). `forType(type: String): List<String>` is case-insensitive on the canonical three types, falls back to `EXPENSE` for unknown/blank (conservative default — falling back to `INCOME` would re-risk the bug C05 fixes).
2. ✅ `TransactionDialog.kt` — replaced the flat `listOf("Food","Rent",...)` with `remember(isIncome) { Categories.INCOME or EXPENSE + "Custom" }`. `LaunchedEffect(isIncome) { selectedCategory = "" }` resets selection on every toggle flip. The hacky special-case "if currently Food → swap to Salary" on the Income button is gone.
3. ✅ `EditTransactionDialog.kt` — routed through `Categories.forType(transaction.type) + "Custom"`. Edit dialogs never change type, so the list is keyed off the existing `transaction.type` field. Bonus: the hardcoded `"Expense"` and `"Balance Correction"` entries (both problematic — the first is C03a's forbidden type-literal, the second is an internal repository-only category) are removed from the user-pickable list.
4. ✅ `BudgetScreen.kt AddBudgetDialog` — refactored to source `Categories.EXPENSE` instead of an inline literal. No behaviour change (Budgets gate outflows only — no Income/Expense toggle present, so no bleed was possible — but consolidating to the shared list keeps category vocabularies in sync as they evolve).
5. ⏸️ **Out of scope (not a C05 bleed):** `RecurringScreen.kt` uses a free-text `OutlinedTextField` for category, not a chip picker, so there's no list to swap. Converting it to a typed chip picker is UX work that belongs under C04 (smart defaults) or a separate enhancement — noted but not done here.

**Test:** `CategoriesDataIntegrityTest` adds 9 pure-function cases covering the core `INCOME ∩ EXPENSE = ∅` invariant plus the audit's acceptance scenarios ("Toggle to Income → see Salary/Freelance/Interest. Toggle to Expense → see Food/Fuel/Shopping/Bills"). Picked up by `checks.yml`'s data-integrity CI gate. Gate count: **30 → 39** (across 5 classes).

**Acceptance:** Toggle to Income → see Salary/Freelance/Interest. Toggle to Expense → see Food/Fuel/Shopping/Bills. Never the other way.

### C08 — Number formatting inconsistency

**Raw points:** 48, 117, 120, 137, 146, 147, 176, 320, 352, 353
**Severity:** UX / polish
**Priority:** P1
**Effort:** M (2-3 days)
**Sprint:** 2

### INF01 — CI workflows red on `master` (Android CI + Fynlo CI Checks)

**Logged:** 2026-05-26 · **Status:** RESOLVED 2026-05-26 (commit `b1e28a0`)
**Severity:** Tooling only — does not affect the shipped app; gated nothing user-facing
**Priority:** P3 (Infrastructure)

Both Android CI workflows failed on every push to `master`. Pre-existing — observed failing on commits `41fb6d4` and `5b42336`, both *before* `checks.yml` was added by the governance work. Adding `checks.yml` introduced no regression to `android.yml`; both were broken for distinct reasons that were only fully diagnosed when the authenticated GitHub Actions log API was queried with the local git credential.

**Root cause — four layered issues, peeled in order:**

| # | Issue | Symptom | Fix |
|---|---|---|---|
| **L1** | `gradle/wrapper/gradle-wrapper.jar` was gitignored (with the comment *"optional — keeps repo lighter"*) | `./gradlew` couldn't bootstrap: `Error: Unable to access jarfile /home/runner/work/Fynlo/Fynlo/gradle/wrapper/gradle-wrapper.jar` | Untracked the wrapper jar from `.gitignore` and committed it (45,457 bytes). The wrapper jar is the version-pinned Gradle bootstrap and is meant to be committed by convention. **Commit `df17aca`.** |
| **L2** | `gradle/actions/setup-gradle@v3` is incompatible with Gradle 9.4.1 — its cache-cleanup init script (`dummy-cleanup-project/init.gradle`) calls `removeUnusedEntriesOlderThan`, which became a write-only property in Gradle 9 | Every `./gradlew` invocation failed in 6s with: `Cannot get the value of write-only property 'removeUnusedEntriesOlderThan' for object of type org.gradle.api.internal.cache.DefaultCacheConfigurations$DefaultCacheResourceConfiguration` | Upgraded `setup-gradle@v3 → @v4` in both workflows (5 occurrences). **Commit `67da7f2`.** |
| **L3** | `gradlew` committed with git mode `100644` (not executable) | Worked locally and in `android.yml` (which has a defensive `chmod +x ./gradlew` step) but would have broken every `checks.yml` job, which doesn't | `git update-index --chmod=+x gradlew` → mode now `100755` on every fresh checkout. **Commit `67da7f2`.** |
| **L4** | Latent: the `dev` flavor's `app/src/dev/google-services.json` is gitignored, and the committed `app/google-services.json` only contains `app.fynlo` (not `app.fynlo.dev`). `assembleDebug` / `test` are aggregate tasks that build **both** flavors, so the Google Services plugin would have failed the dev variant with *"No matching client found for package name app.fynlo.dev"* | Never actually surfaced in CI because L1 fired first, but it *would have* been the next failure if the wrapper jar alone had been fixed | Scoped CI to the `prod` flavor (whose `google-services.json` is committed). **Commit `05a696f`.** |

**Diagnostic mis-steps along the way (logged so future me doesn't repeat them):**
- The first hypothesis attributed the failure to L4 alone, based on reasoning about flavors and the `.gitignore` for `google-services.json`. That hypothesis was wrong as the *immediate* cause — confirmed only when the CI job log (fetched via the stored git credential against the authenticated GitHub API) showed `Unable to access jarfile`, which is L1 firing before anything else runs.
- A local `assembleProdDebug` run reported `BUILD SUCCESSFUL` but the compile tasks were `UP-TO-DATE` from Android Studio's cache — *did not* exercise a clean compile. Lesson: for "does this build in CI?" validation, run with `--rerun-tasks` or in a clean clone, not against an IDE-warmed cache.

**Resolution summary (commits in landing order):**

1. **`05a696f`** — scoped CI to `prod` flavor (`assembleProdDebug`, `testProdDebugUnitTest`, `assembleProdRelease`); added Android Lint baseline (INF02); removed unconfigured Detekt step (INF03); removed "No tests found" data-integrity filter step (INF04).
2. **`df17aca`** — committed `gradle/wrapper/gradle-wrapper.jar`, removed it from `.gitignore`, slimmed `android.yml`'s release job to unsigned verification (`./gradlew :app:assembleProdRelease`, no signing/secrets/env-vars/artifact-upload).
3. **`67da7f2`** — `setup-gradle@v3 → @v4` in both workflows; `gradlew` git mode `100755`.
4. **`b1e28a0`** — added the C01 Recalculate Room regression test (this is the test step 5 of C01 calls for; `@Ignore`'d pending the fix — see INF04).

**Verified green** on commit `b1e28a0`: Android CI ✅ (Build & Test ✅, Release build verify ✅), Fynlo CI Checks ✅ (Lint ✅, Unit tests ✅, Build release ✅), Pages ✅.

### INF05 — Instrumented tests don't run in CI

**Logged:** 2026-05-26 · **Priority:** P2 (Infrastructure — every future schema change should be gated by this)

`FynloDatabaseMigrationTest` (`app/src/androidTest/java/app/fynlo/data/local/`) is the canonical Room migration test for the `v15 → v16` C01 backfill — five cases, all passing on a real device (CPH2767 / Android 16, 2026-05-26). But because it lives in `androidTest/` it requires the Android instrumentation harness, which neither of the two Gradle CI workflows (`android.yml`, `checks.yml`) currently provides. The test ran locally before tagging 3.2.2 (as the third gate in `RELEASE_PROTOCOL.md §8`), and it has to be re-run *manually* on every future schema-change PR until this item is fixed.

Three options for closing this:

1. **Firebase Test Lab integration.** Free tier covers ~5 device-minutes per workflow run, enough for a focused `--tests '*MigrationTest*'` invocation. Adds a `gcloud auth` secret + the `firebase test android run` step to `android.yml`. Lowest infra burden; modest paid-tier cost if the test suite grows.
2. **GitHub-hosted runner with a managed emulator.** `reactivecircus/android-emulator-runner@v2` action. Works on the same `ubuntu-latest` runner we already use, no external service. Slower per-run (cold emulator boot ~3 min on top of build time).
3. **Self-hosted runner with a persistent emulator/device.** Lowest per-run cost; highest initial setup. Only justified if instrumented testing volume grows substantially.

Recommended start: option 2 (`reactivecircus/android-emulator-runner`) — bounded scope, no new secrets needed, no cloud account dependency. Move to option 1 if emulator boot time becomes the bottleneck.

Trigger pattern (any of these touches schema/migrations code → run instrumented tests):

```yaml
paths:
  - 'app/src/main/java/app/fynlo/data/local/FynloDatabase.kt'
  - 'app/src/main/java/app/fynlo/data/local/FynloDao.kt'
  - 'app/src/main/java/app/fynlo/data/model/**'
  - 'app/schemas/**'
  - 'app/src/androidTest/**'
```

Until then: any PR that adds a `MIGRATION_<N>_<N+1>` must run `./gradlew :app:connectedProdDebugAndroidTest --tests '*MigrationTest*'` locally before merge. Mention this rule in the PR template if it gets missed.

### INF06 — `ScrollBenchmark` CUJ is brittle on fresh-install device state

**Logged:** 2026-05-26 · **Priority:** P3 (Infrastructure — `RELEASE_PROTOCOL §3.4` scroll bullet is currently un-runnable without manual setup)

Observed during the 3.2.2 release-prep macrobench: after `adb uninstall app.fynlo` (forced by the install-signature mismatch with debug-signed installs), the `:macrobenchmark:connectedProdBenchmarkBenchmarkAndroidTest` *startup* run worked cleanly, but the *scroll* run reported **`frameCount = 1.0` across all 8 iterations** (vs `1061` in the baseline). The device was left on `com.android.contacts/.DialtactsActivityAlias` — the **system** Contacts app — not Fynlo's `PeopleScreen`.

**Root cause (most likely):** the setup chain in `ScrollBenchmark.scrollPeople`'s `setupBlock`:

```kotlin
pressHome()
startActivityAndWait()
skipFirstLaunch()                                       // Skip × 2 + "Continue without signing in"
device.wait(Until.findObject(By.desc("Menu")), 5_000)?.click()
device.wait(Until.findObject(By.text("Contact Book")), 5_000)?.click()
```

falls over in two ways on a freshly-installed app:

1. **`skipFirstLaunch()` is best-effort** — it `?.click()`s null-safely on each step, so if a "Skip" doesn't appear at the expected pace (slow first-launch, animation in progress, KSP-cold pipeline), the function exits early with the app stuck on onboarding or the setup wizard.
2. **`By.text("Contact Book")` is globally scoped on UiAutomator** — if Fynlo's drawer never opened, the matcher cascades to the *launcher's* Contact Book tile (the system Contacts app on Oppo / CPH2767), which gets clicked and shifts focus away from Fynlo entirely. The subsequent scroll happens on an empty Contacts list → ~1 frame per iteration.

**Fix candidates (any one resolves the failure mode):**

- **A. Make `skipFirstLaunch()` validating, not best-effort.** After each `?.click()`, `device.wait(Until.findObject(By.text("<next-screen-marker>")), 10_000)` and fail loudly if not found. Catches the silent skip.
- **B. Scope the Contact Book lookup to the target package.** `By.pkg(TARGET_PACKAGE).text("Contact Book")` — refuses to match the system Contacts app icon on the launcher.
- **C. Tear-down + manual setup before scroll bench.** Document a pre-run step that completes onboarding + setup wizard once before the gradle invocation; subsequent iterations reuse the persisted DataStore flag. Lowest code change; highest friction for the human running the bench.

Recommended: **B** as the primary fix (one-line change, refuses to leave the target app), with **A** as defensive hardening on top. The setup-wizard PIN step in particular needs A's validating behaviour because it can take a beat to render.

**Until this is fixed**, the scroll benchmark cannot reliably gate releases. The 3.2.2 release was promoted on the basis of startup-green (`§3.4` bullet 1) and unchanged-UI surface (no `PeopleScreen` / `Composable` touched in 3.2.0 → 3.2.2), with `§3.4` bullet 2 explicitly deferred to a release that includes actual UI changes — at which point INF06 needs to land first.
