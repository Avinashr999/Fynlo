## 1. The one finding that overrides everything

**Recalculate Balances destroys user data.**

When tapped, it deletes the `paid` field from every borrower and debt record. Ôé╣54,000 of repayment history evaporated in the audit's test data. The XLSX export's "Loan Repayments" and "Debt Repayments" sheets are always empty because Fynlo doesn't store individual payments as records ÔÇö only as a cumulative `paid` field on the parent. Wiping the field wipes the history.

This is **C01** in the cluster list and **P0-blocker #1** in the priority list.

**Recommendation: Do NOT ship 3.2.1 to Play Store production until C01 is fixed.**

The cold-start performance win from the baseline profile is real (~10% faster) and worth shipping ÔÇö but not alongside a regression that nukes payment history when users tap a button labeled "Fixes account balances that got out of sync."

Ship 3.2.1 via Play Store internal testing only. Land the C01 fix as 3.2.2 and promote that to production instead.

## 2. The 22 root-cause clusters

### C01 ÔÇö Recalculate Balances data destruction

**Raw points:** 346, 347, 348, 349, 350, 367
**Severity:** Critical data loss
**Priority:** P0 (ship-blocker)
**Effort:** L (3-5 days)
**Sprint:** 1

**Root cause:**
1. Payments are stored as a cumulative `paid: Double` field on the parent loan/debt record, not as individual payment records.
2. The Recalculate flow rebuilds loan/debt state from interest accrual without preserving the `paid` field.
3. `updatedAt` timestamps don't update when the destruction happens, breaking sync conflict resolution.

**Fix:**
1. **Schema change:** Add `loanRepayments[]` and `debtRepayments[]` collections. Each repayment is a record with `id`, `parentLoanId`/`parentDebtId`, `amount`, `date`, `note`, `createdAt`. Derive cumulative `paid` from the sum.
2. **Migration:** Backfill existing `paid` fields into single repayment records dated `loanDate` (best available approximation).
3. **Recalculate rewrite:** Compute outstanding = principal + accrued interest Ôłĺ sum(repayments). Never touch repayment records.
4. **`updatedAt` correctness:** Update timestamp on every actual mutation.
5. **Guard:** Add a unit test that runs Recalculate against a fixture with known `paid` values and asserts the sums are preserved.

**Impact on other clusters:** Unblocks C02 (stale exports), C03 (sync correctness).

### C02 ÔÇö Stale exports & no auto-recalc

**Raw points:** 331, 332, 333, 334, 343, 344, 345
**Severity:** Data integrity
**Priority:** P0
**Effort:** M (2-3 days)
**Sprint:** 1

**Root cause:** Recalculate is manual-only. PDF, JSON, and XLSX exports use whatever in-memory state exists, which may be days or weeks stale. The PDF generated pre-recalc showed Ôé╣268,081 net worth; post-recalc showed Ôé╣241,663 ÔÇö same data, Ôé╣26K difference, no warning.

**Fix:**
1. **Auto-recalc on app launch.** Run once per day max (debounce on `lastRecalcAt < today.startOfDay`).
2. **Auto-recalc before every export.** PDF, JSON, XLSX must invoke recalc as a pre-step.
3. **Surface `lastRecalcAt` in UI.** Small subtitle on Dashboard near Net Worth: "Last updated 2 min ago".
4. **Show before/after on manual recalc.** When user taps Recalculate in Settings, show a result dialog: "Net worth: Ôé╣268,081 Ôćĺ Ôé╣241,663. Interest accrued: Ôé╣85,651."
5. **Embed timestamp in exports.** PDF header includes "Recalculated: 25 May 2026, 9:07 AM."

**Dependency:** Requires C01 (otherwise auto-recalc destroys data daily).

**Stage 1 landed 2026-05-26 (fixes #1, #2 of the five):**

- New `app.fynlo.data.RecalcCoordinator` (`@Singleton`, Hilt-injected) wraps `FinanceRepository.recalculateAllBalances()` and stamps `UserPreferences.lastRecalcAt` after each successful run. Exposes a pure `shouldRecalcOnLaunch(lastRecalcAt, now, zone)` predicate on the companion (testable without Robolectric ÔÇö calendar-day boundary in the user's local zone, NOT a 24h sliding window).
- `FynloApplication.onCreate` post-init coroutine now calls `recalcCoordinator.runIfStaleOnLaunch()` ÔÇö fire-and-forget; failure is non-fatal.
- `FinanceViewModel` exports (`exportToCSV`, `exportToPDF`, `exportAllData`, plus a NEW `exportToXLSX`) all call `recalcCoordinator.runAndStamp()` before generating output. XLSX was previously skipped because the launcher called `ExcelExportUtility.generateFullBackup` directly ÔÇö now routed through the ViewModel for consistency.
- `SettingsScreen` PDF launcher wrapped in `scope.launch(Dispatchers.IO)` (since `exportToPDF` is now suspend).
- `AutoRecalcDataIntegrityTest` (8 cases) covers the debounce predicate at every interesting edge: first-launch (lastRecalcAt = 0), midnight rollover, same-day-twice, local-vs-UTC zone, week-long gap, exact-equality-with-now, and a from-first-principles arithmetic check against `LocalDate.atStartOfDay`. Picked up by the `checks.yml` data-integrity gate (matches `*DataIntegrity*` and `*Recalculate*`).

**Stage 2 landed 2026-05-26 (fixes #3, #4, #5 ÔÇö C02 fully CLOSED):**

- **Dashboard "Last updated X ago" subtitle** ÔÇö `HomeScreenModern` consumes `UserPreferences.lastRecalcAt(...).collectAsState(initial = 0L)` and renders below the hero net-worth number using Android's `DateUtils.getRelativeTimeSpanString` (yields "2 min ago" / "yesterday" / "3 days ago" with appropriate granularity). Shows "Not recalculated yet" when `lastRecalcAt == 0L`.
- **Before/after AlertDialog on manual Recalculate** ÔÇö `FinanceViewModel.recalculateAllBalancesCapturingDelta()` captures `financialSummary.value` before, suspending-calls `runAndStamp()`, then waits up to 500 ms for the next `FinancialSummary` emission different from the pre-snapshot (`withTimeoutOrNull`). Returns a `RecalcDelta` data class with `netWorthChange` / `receivablesChange` / `cashChange` / `investmentsChange` + an `isNoOp` predicate. `SettingsScreen`'s recalc handler shows the dialog with `before Ôćĺ after` for net worth plus signed deltas for the other three; if `isNoOp` (the common post-C01 case), the dialog shows a "your data was already up to date" message instead. The old fire-and-forget Toast is gone.
- **Timestamp in exports** ÔÇö `ExportUtility.generatePDF` and `ExcelExportUtility.generateFullBackup` both accept a new `lastRecalcAt: Long = 0L` parameter. PDF: a `Recalculated: <date>` line under the existing `Generated:` line (or `ÔÇö` if `lastRecalcAt == 0L`). XLSX: a new top-of-file `Metadata` sheet with `Generated` / `Recalculated at` / `Export type` rows. `RecalcCoordinator.runAndStamp()` was refactored to **return** the stamped time so the ViewModel exports don't need a separate DataStore read.

**Net effect:** the audit's quoted scenario ("PDF generated pre-recalc showed Ôé╣268,081; post-recalc showed Ôé╣241,663 ÔÇö same data, Ôé╣26K difference, no warning") is no longer reachable. The PDF can't show stale numbers because exports recalc first, the dashboard tells you when the data was computed, and tapping Recalculate explicitly shows the delta instead of silently mutating things.

### C03 ÔÇö Schema integrity & sync correctness

**Raw points:** 276, 277, 278, 279, 280, 281, 282, 283, 284, 285, 286, 287, 289, 290, 291, 296, 297, 301
**Severity:** Data integrity
**Priority:** P0 / P1 mix
**Effort:** XL (1-2 weeks across the cluster)
**Sprint:** 2-3

**Root cause:** The data layer was built incrementally without a schema discipline. As a result:
- No `schemaVersion` / `appVersion` / `exportedAt` at backup root Ôćĺ restore can't migrate forward.
- Transactions reference accounts by `name` not `id` Ôćĺ renaming "Cash in Hand" breaks every historical transaction.
- `updatedAt` is epoch ms; `date` is ISO string Ôćĺ inconsistent within same record.
- Borrowers use `type`, Debts use `intType` Ôćĺ same concept, two field names.
- No `projectId` anywhere Ôćĺ "Manage Projects" feature has nothing to scope.
- Borrower phone is free-text, not FK to a `people[]` record Ôćĺ same person stored 3x for 3 loans.
- One transaction has `category: "Expense"` literal Ôćĺ default category bug.
- No `createdAt` field anywhere Ôćĺ can't tell when records originated.
- No data integrity hash / checksum Ôćĺ corrupted file restores silently.
- Mixed ID strategy (`acc-cash` vs UUIDs) Ôćĺ collision risk on multi-device merge.

**Fix ÔÇö staged:**

**Stage 3a (Sprint 2) ÔÇö additive, non-breaking: Ôťů FULLY CLOSED 2026-05-26.**

1. Ôťů **Done 2026-05-26 (Stage 1).** Backup root carries `schemaVersion = 2`, `appVersion = BuildConfig.VERSION_NAME`, `exportedAt = Instant.now().toString()`, `userId` (caller-provided, blank if signed out), `deviceName = Build.MODEL`. All five fields are populated by `FinanceRepository.getAllDataAsJson()`.
2. Ôťů **Done 2026-05-26 (Stage 2).** `MIGRATION_16_17` adds `createdAt INTEGER NOT NULL DEFAULT 0` to `flow_templates`, `investment_valuations`, `net_worth_snapshots`, `recurring_transactions`. Three of the four are backfilled from `updatedAt`; `net_worth_snapshots` (no `updatedAt`) is backfilled from `strftime('%s', date) * 1000`. Entity classes updated to add the field.
3. Ôťů **Done 2026-05-26 (Stage 2).** `MIGRATION_16_17` adds `projectId TEXT NOT NULL DEFAULT 'personal'` to `investment_valuations`, then backfills from the parent investment's `projectId` (orphan valuations fall back to `'personal'`).
4. Ôťů **Done 2026-05-26 (Stage 1).** `contentHash` is a SHA-256 (lower-case hex) over the canonical JSON form of `BackupData` with `contentHash` set to `""`. Computed at export, embedded as the last step before serializing. Verified at import via `BackupIntegrity.check()` ÔÇö restore refuses to proceed on hash mismatch (`IllegalStateException` thrown before `db.withTransaction` opens). Forward-compat: backups with `schemaVersion > 2` are also refused with `UnsupportedVersion`. Backwards-compat: v1 legacy backups (no metadata, no hash) are accepted unconditionally. `BackupDataIntegrityTest` covers all 10 paths.
5. Ôťů **Done 2026-05-26 (Stage 2).** NEW `app.fynlo.data.TransactionValidator.sanitize(Transaction)` rewrites the three forbidden literal categories (`"Expense"` / `"Income"` / `"Transfer"`) to `"Uncategorized"`. Applied at both write sites: `FinanceRepository.insertTransaction` and the new-side of `editTransaction`. Matching is case-sensitive and exact ÔÇö `"Income Tax"` and `"Expense Reimbursement"` pass through unchanged. A one-shot `UPDATE transactions SET category = 'Uncategorized' WHERE category IN ('Expense', 'Income', 'Transfer')` inside `MIGRATION_16_17` fixes any existing rows produced by the old buggy dropdown. `TransactionValidatorDataIntegrityTest` covers the contract (9 pure-function cases including idempotency, exact-match-only, fast-path no-allocation when input is valid).

**Sprint 1+2 milestone:** With C01 + C02 + C03a all closed, three of the four P0 ship-blockers (C01 / C02 / C03a) are done. The Sprint 1 audit estimate was C01-only; we landed C01 + C02 + C03a in the same window. C05 (category bleed Income/Expense) is the one remaining P0.

**Sprint 2+ remaining:** Stage 3b (breaking migrations ÔÇö account-name Ôćĺ account-ID, `intType` unification, peopleId/contact linkage, UUID standardization) is unchanged from the original audit plan ÔÇö Stage 3a's additive groundwork doesn't unblock or unblock Stage 3b directly.

**Stage 3b (Sprint 3) ÔÇö breaking, requires migration:**
1. Convert `txn.fromAcct` / `txn.toAcct` from name strings to account IDs. Migration: lookup by name on import, store ID going forward.
2. Standardize `intType` everywhere (rename `borrowers[].type` to `borrowers[].intType`).
3. Move `borrower.phone` to a separate `people[]` record. Borrower references `peopleId`. Migration: dedupe phones, create people records, repoint.
4. Standardize on UUIDs across all collections. Migration: `acc-cash` Ôćĺ new UUID, update all references.

Each stage ships independently. Stage 3a is fully backwards-compatible. Stage 3b requires a versioned migration script that runs once at upgrade.

**Acceptance:** Round-trip test ÔÇö export, wipe, restore, diff. Result must be byte-identical for all post-stage-3b backups.

### C04 ÔÇö Smart defaults system (the missing system) Ôťů CLOSED 2026-05-27 in 3.2.6 (smoke-follow-up patch in 3.2.7)

**Raw points:** 14, 15, 16, 167, 168, 213, 215, 216, 229, 230, 237, 263
**Severity:** UX friction, daily
**Priority:** P1
**Effort:** M (3-5 days) ÔÇö actual: ~1 session (Stages 1+2 plus Stage 3 + 2.5 closure)
**Sprint:** 2 (first P1 cluster closed)

**Root cause:** Every form starts blank or with a wrong default. User taps + 30 times a day. Multiplied across forms (Add Transaction, Add Recurring, Add Savings, Set Category Limit, currency picker, account picker), this is the single biggest source of friction.

Specific missing defaults:
- Add Transaction: no default category (or wrong category, see C05). Should default to user's most-recently-used category for the current type.
- Add Transaction: no default account. Should default to most-recently-used.
- Add Recurring: no default category, no default account. Same fix.
- Set Category Limit: no default category. Should default to user's highest-spend uncapped category.
- Settings Ôćĺ currency: defaults to INR regardless of locale. Should default to system locale on first install.
- Recently-used should also appear at top of pickers (currency picker has no "recent" group).

**Fix:**
1. Build a `RecentlyUsedTracker` with per-form, per-field memory.
2. On every form open, prefill the field with `recentlyUsed.last(formId, fieldId)` or fall back to system locale / heuristic.
3. Picker dropdowns sort: recently-used (top, ÔëĄ5 items) Ôćĺ alphabetical (everything else).
4. Migrate to using `system_locale_default` for currency on fresh install.

- NEW `app.fynlo.data.RecentlyUsedLogic` ÔÇö pure-function helpers (`add` / `last` / `topN` / `keyOf`). All dedup-and-cap semantics live here, independent of DataStore, so unit tests cover the contracts without Robolectric. `MAX_ENTRIES = 5` per the audit's "ÔëĄ5 items" group size.
- NEW `app.fynlo.data.model.RecentlyUsedSnapshot` / `RecentEntry` ÔÇö serializable types stored as a single JSON blob under `UserPreferences.RECENTLY_USED`. Single Preferences key keeps the dataset (ÔëĄ5 entries ├Ś ~10 slots) addressable without a new Room table or migration. Forward-compat is the kotlinx-serialization default-values story.
- `UserPreferences.recentlyUsed(ctx): Flow<RecentlyUsedSnapshot>` + `editRecentlyUsed(ctx, mutate)` ÔÇö DataStore reader / atomic editor. Fault-tolerant decode: a corrupted blob yields an empty snapshot instead of crashing.
- `RecentlyUsedDataIntegrityTest` ÔÇö 12 pure-function cases. Picked up by the `*DataIntegrity*` CI gate filter. Gate count: **39 Ôćĺ 51** across 6 classes.

### INF05 ÔÇö Instrumented tests don't run in CI

**Logged:** 2026-05-26 ┬Ě **Priority:** P2 (Infrastructure ÔÇö every future schema change should be gated by this)

`FynloDatabaseMigrationTest` (`app/src/androidTest/java/app/fynlo/data/local/`) is the canonical Room migration test for the `v15 Ôćĺ v16` C01 backfill ÔÇö five cases, all passing on a real device (CPH2767 / Android 16, 2026-05-26). But because it lives in `androidTest/` it requires the Android instrumentation harness, which neither of the two Gradle CI workflows (`android.yml`, `checks.yml`) currently provides. The test ran locally before tagging 3.2.2 (as the third gate in `RELEASE_PROTOCOL.md ┬ž8`), and it has to be re-run *manually* on every future schema-change PR until this item is fixed.

Three options for closing this:

1. **Firebase Test Lab integration.** Free tier covers ~5 device-minutes per workflow run, enough for a focused `--tests '*MigrationTest*'` invocation. Adds a `gcloud auth` secret + the `firebase test android run` step to `android.yml`. Lowest infra burden; modest paid-tier cost if the test suite grows.
2. **GitHub-hosted runner with a managed emulator.** `reactivecircus/android-emulator-runner@v2` action. Works on the same `ubuntu-latest` runner we already use, no external service. Slower per-run (cold emulator boot ~3 min on top of build time).
3. **Self-hosted runner with a persistent emulator/device.** Lowest per-run cost; highest initial setup. Only justified if instrumented testing volume grows substantially.

Recommended start: option 2 (`reactivecircus/android-emulator-runner`) ÔÇö bounded scope, no new secrets needed, no cloud account dependency. Move to option 1 if emulator boot time becomes the bottleneck.

Trigger pattern (any of these touches schema/migrations code Ôćĺ run instrumented tests):

```yaml
paths:
  - 'app/src/main/java/app/fynlo/data/local/FynloDatabase.kt'
  - 'app/src/main/java/app/fynlo/data/local/FynloDao.kt'
  - 'app/src/main/java/app/fynlo/data/model/**'
  - 'app/schemas/**'
  - 'app/src/androidTest/**'
```

Until then: any PR that adds a `MIGRATION_<N>_<N+1>` must run `./gradlew :app:connectedProdDebugAndroidTest --tests '*MigrationTest*'` locally before merge. Mention this rule in the PR template if it gets missed.

### INF06 ÔÇö `ScrollBenchmark` CUJ is brittle on fresh-install device state

**Logged:** 2026-05-26 ┬Ě **Priority:** P3 (Infrastructure ÔÇö `RELEASE_PROTOCOL ┬ž3.4` scroll bullet is currently un-runnable without manual setup)

Observed during the 3.2.2 release-prep macrobench: after `adb uninstall app.fynlo` (forced by the install-signature mismatch with debug-signed installs), the `:macrobenchmark:connectedProdBenchmarkBenchmarkAndroidTest` *startup* run worked cleanly, but the *scroll* run reported **`frameCount = 1.0` across all 8 iterations** (vs `1061` in the baseline). The device was left on `com.android.contacts/.DialtactsActivityAlias` ÔÇö the **system** Contacts app ÔÇö not Fynlo's `PeopleScreen`.

**Root cause (most likely):** the setup chain in `ScrollBenchmark.scrollPeople`'s `setupBlock`:

```kotlin
pressHome()
startActivityAndWait()
skipFirstLaunch()                                       // Skip ├Ś 2 + "Continue without signing in"
device.wait(Until.findObject(By.desc("Menu")), 5_000)?.click()
device.wait(Until.findObject(By.text("Contact Book")), 5_000)?.click()
```

falls over in two ways on a freshly-installed app:

1. **`skipFirstLaunch()` is best-effort** ÔÇö it `?.click()`s null-safely on each step, so if a "Skip" doesn't appear at the expected pace (slow first-launch, animation in progress, KSP-cold pipeline), the function exits early with the app stuck on onboarding or the setup wizard.
2. **`By.text("Contact Book")` is globally scoped on UiAutomator** ÔÇö if Fynlo's drawer never opened, the matcher cascades to the *launcher's* Contact Book tile (the system Contacts app on Oppo / CPH2767), which gets clicked and shifts focus away from Fynlo entirely. The subsequent scroll happens on an empty Contacts list Ôćĺ ~1 frame per iteration.

**Fix candidates (any one resolves the failure mode):**

- **A. Make `skipFirstLaunch()` validating, not best-effort.** After each `?.click()`, `device.wait(Until.findObject(By.text("<next-screen-marker>")), 10_000)` and fail loudly if not found. Catches the silent skip.
- **B. Scope the Contact Book lookup to the target package.** `By.pkg(TARGET_PACKAGE).text("Contact Book")` ÔÇö refuses to match the system Contacts app icon on the launcher.
- **C. Tear-down + manual setup before scroll bench.** Document a pre-run step that completes onboarding + setup wizard once before the gradle invocation; subsequent iterations reuse the persisted DataStore flag. Lowest code change; highest friction for the human running the bench.

Recommended: **B** as the primary fix (one-line change, refuses to leave the target app), with **A** as defensive hardening on top. The setup-wizard PIN step in particular needs A's validating behaviour because it can take a beat to render.

**Until this is fixed**, the scroll benchmark cannot reliably gate releases. The 3.2.2 release was promoted on the basis of startup-green (`┬ž3.4` bullet 1) and unchanged-UI surface (no `PeopleScreen` / `Composable` touched in 3.2.0 Ôćĺ 3.2.2), with `┬ž3.4` bullet 2 explicitly deferred to a release that includes actual UI changes ÔÇö at which point INF06 needs to land first.
